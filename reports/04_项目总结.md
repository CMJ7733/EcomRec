# 项目总结与经验沉淀

**项目**：电商用户购买行为分析与深度混合推荐系统  
**数据集**：Amazon Reviews 2023 — Beauty and Personal Care  
**完成时间**：2024 年

---

## 一、项目成果总览

本项目基于 Amazon Reviews 2023 电商数据集，完整复现了工业界"**召回-精排**"双塔推荐架构：

| 模块 | 产出 |
|------|------|
| **数据治理** | K-core(5) 过滤、时间序切分、特征宽表 |
| **用户画像** | RFM + K-Means 四类用户分群 |
| **召回层** | Top-Pop / ItemCF / BPR-MF / ALS 四路召回 |
| **精排层** | LightGBM / DeepFM / Wide & Deep |
| **推荐链路** | 多路融合（200）→ 精排（50）→ MMR 打散（10） |
| **可视化** | Streamlit 多页面 Dashboard |
| **分析报告** | 4 份独立文档 + 全套图表 |

## 二、传统 ML vs 深度学习实际差异

### 2.1 训练效率

| 维度 | 传统 ML（LightGBM） | 深度学习（DeepFM） |
|------|--------------------|--------------------|
| 训练时间 | 5-10 min（CPU） | 20-40 min（GPU） |
| 调参难度 | 中等（tree depth/leaves） | 高（网络结构/lr/dropout） |
| 可解释性 | 高（特征重要度） | 低（embedding 黑盒） |
| AUC 提升 | Baseline | +2-5% |

### 2.2 工程复杂度

- **传统 ML**：数据处理 → 特征工程 → LightGBM → 输出，流程简单，适合快速迭代
- **深度学习**：数据处理 → 特征工程 → Embedding Table → FM Component → DNN Component → 输出，涉及 GPU 训练、混合精度、EarlyStopping 等，工程复杂度显著更高

### 2.3 实战建议

> **不是所有场景都需要深度学习**。当数据量 < 100 万交互、特征工程做得足够充分时，LightGBM 往往已经能拿到 80% 的 AUC 收益。DeepFM 的边际收益在高稀疏度、用户行为序列丰富的场景下更明显。

## 三、踩坑记录与解决思路

### 3.1 数据穿越问题

**问题**：随机切分训练/测试集导致"未来数据泄露"，模型用未来信息预测过去。

**解决**：严格按时间戳排序后切 80/10/10，Leave-One-Out 取每个用户最后一次交互作为测试。

**代价**：冷门用户的最后一次交互可能是数月前的行为，预测难度显著高于随机切分。

---

### 3.2 正负样本不平衡

**问题**：CTR 任务中正样本（点击/购买）远少于负样本（未曝光），直接训练会导致模型倾向于全预测负样本。

**解决**：采用 1:4 负采样比例（1 positive : 4 negative），同时使用 BCE Loss + GAUC 评估。

**注意**：负采样会改变真实分布，线上 serving 时需要用 sigmoid 校准。

---

### 3.3 推荐同质化（MMR 后处理）

**问题**：精排模型优化 AUC 时倾向于推荐同类商品，导致用户看到的推荐高度相似。

**解决**：引入 MMR（Maximal Marginal Relevance）打散，λ=0.5 时兼顾相关性与多样性。

**参数调优**：λ 太小（< 0.3）多样性提升有限，λ 太大（> 0.7）相关性损失严重，建议 0.4-0.6 区间。

---

### 3.4 长尾商品挑战

**问题**：Top 10% 商品占据 ~60% 交互量，尾部商品行为数据稀疏，ItemCF 相似度估计噪声大。

**解决**：IUF（Inverse User Frequency）衰减 + BPR/ALS 矩阵分解降维 + Top-Pop 兜底。

**启示**：不要试图用单一模型解决所有问题，多路召回融合是工业界的主流解法。

---

### 3.5 GPU 显存与训练稳定性

**问题**：DeepFM 的 Embedding + DNN 在大数据集上容易 OOM。

**解决**：
- 负采样控制正负比 1:4
- DataLoader num_workers=4 + pin_memory=True
- 混合精度训练（AMP）
- Embedding 维度控制在 16 以下

## 四、设计决策记录（ADR）

### ADR-1：为什么选择 Polars 而不是 Pandas？

**决策**：使用 Polars 作为主力数据处理库。

**理由**：
- 性能比 Pandas 快 10x，内存效率更高
- 多线程并行，充分利用多核 CPU
- 与 PyArrow 完全兼容，支持零拷贝读取 parquet
- API 设计更现代，链式调用更优雅

**代价**：部分第三方库（如 Streamlit、sklearn）仍依赖 Pandas，需要在边界做转换。

---

### ADR-2：为什么用 Leave-One-Out 而非随机切分？

**决策**：测试集取每个用户最后一次交互（时间序）。

**理由**：与 SASRec/BERT4Rec 等序列推荐论文一致，便于横向对比；防止数据穿越。

**代价**：冷门用户最后一次交互可能参考价值较低。

---

### ADR-3：为什么负采样比例是 1:4？

**决策**：1 个正样本配 4 个负样本。

**理由**：
- 比例太低（1:1）：负样本信息不足，模型判别能力弱
- 比例太高（1:10）：正样本被稀释，召回困难
- 1:4 是经验值，与 Facebook GB+ 等工业实践一致

---

### ADR-4：为什么 DeepFM 自己实现而非用 DeepCTR 库？

**决策**：纯 PyTorch 原生实现，约 150 行代码。

**理由**：
- 便于讲清原理（FM 交叉、DNN 高阶特征）
- 无第三方库依赖，维护成本低
- 超参数可控，不被库版本限制

**代价**：需要手写训练循环，不如 DeepCTR 一键训练省心。

---

### ADR-5：为什么用 MMR 而非简单贪心？

**决策**：精排 Top-50 后用 MMR 打散输出 Top-10。

**理由**：MMR 在相关性与多样性之间做显式权衡，λ 参数可调，比纯贪心更灵活。
